异想天开的分3段快排。。。

Aug 24th, 2009

(hplonline)2009.8.20

看了一份研究报告之后，
也手痒了，想想能不能把快排变得更快一点。

之前讨论过STL中一个尽量避免爆栈的写法。
这里突然想到的是改变一下分段数有没有作用。

理论分析：

首先说说一般的快排效率，
我更倾向于立足某个程序实现，分析实际的比较和移动，
而不是单纯的看个理论的复杂度。

比如下方程序中的qsort2就是市面上常见的经典写法。
应该说，已经做得很精炼了，而且结构上看着很整齐。
每次比较的时候区间[l,r]中的每个数都和轴比较了一次。
而当把整个区间分成m段时，每一段比较的是n/m次，
总共加起来还是n次。
也就是说，每次把区间分成两段，不断分下去，形成一棵树。
这棵树的每层耗掉的比较次数都是n次。
而这棵树总共只有log2(n)这么高。
所以总共的耗费是f1=n*log2(n)

qsort3就是后面突发奇想的，
如果每次分成三段的话，
效率会怎么样呢。。
沿用上面的这种分析法。
分成三段，则需要两个轴元素，每个元素进行两次比较。
所以每层是2*n次比较，而总共的层数是log3(n)。
乘起来就是f2=2*n*log3(n)

比较两个式子可以简单使用作商法：
f1/f2
=(n*log2(n))/(2*n*log3(n))
=(ln(n)/ln(2))/(2*ln(n)/ln(3))
=ln(3)/(2*ln(2))
= 0.7925 < 1

也就是说，单看比较次数，
这个想法就很作废了。
不过我还是写了下程序看看。。。

程序：

#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#include <string.h>
#include <algorithm>

using namespace std ;

const int MAXN = 1000000;

int origin[MAXN] , standard[MAXN];
int a[MAXN] ;
int tmp[MAXN] ;

void init(){
    int i;
    for ( i = 0 ; i < MAXN ; i ++ ){
        origin[i] = rand() % 1000 ;
    }
    memcpy(standard , origin , sizeof(origin)) ;
    sort(standard , standard + MAXN) ;
}

void qsort2(int l , int r){
    int i , j , t ;
    i = rand() % ( r - l + 1 ) + l ;
    t = a[i] ;
    a[i] = a[l] ;
    a[l] = t ;
   
    i = l ;
    j = r ;
    //t == a[i]
    while ( i < j ){
        while ( i < j && t <= a[j] ) j – ;
        a[i] = a[j] ;
        while ( i < j && a[i] <= t ) i ++ ;
        a[j] = a[i] ;
    }
    a[i] = t ;

    if ( i - 1 > l ) qsort2(l , i - 1) ;
    if ( i + 1 < r ) qsort2(i + 1 , r) ;
}

void qsort3(int l , int r){
    int k , i , i1 , j , j1 , t1 , t2 ;
   
    t1 = a[l] ;
    if ( t1 > a[l + 1] ) {
        t2 = t1 ;
        t1 = a[l + 1] ;
    }else t2 = a[l + 1] ;

    i = l ;
    j = r ;
    for ( k = l + 2 ; k <= r ; k ++ ){
        if ( a[k] < t1 ) tmp[i ++] = a[k] ;
        else tmp[j –] = a[k] ;
    }

    memcpy(&a[l] , &tmp[l] , sizeof(int) * (i - l)) ;
    i1 = i ;
    a[i ++] = t1 ;

    j1 = j ;
    j = r ;
    for ( k = j1 + 1 ; k <= r ; k ++ ){
        if ( tmp[k] > t2 ) a[j –] = tmp[k] ;
        else a[i ++] = tmp[k] ;
    }
    a[j] = t2 ;
   
    if ( i1 - 1 > l ) qsort3(l , i1 - 1) ;
    if ( j - i1 > 2 ) qsort3(i1 + 1 , j - 1) ;
    if ( j + 1 < r ) qsort3(j + 1 , r) ;

}

void test(void (*func)(int , int)){
    memcpy(a , origin , sizeof(origin)) ;
    int t = clock() ;
    (*func)(0 , MAXN - 1) ;
    if ( memcmp(standard , a , sizeof(origin)) == 0 ) printf(“right:”) ;
    else printf(“wrong:”) ;
    printf(“%d\n” , clock() - t) ;
}

int main(){
    init() ;
    test(qsort2) ;
    test(qsort3) ;
    return 0 ;
}

结果：（debug编译）

right:1576
right:3010

结论是很显然的，效率没有改进。
而且慢得比理论推算的比例更多。。

最后再认真想一下，
发现最初的想法忽略的最重要的一个问题，
就是分段的均匀性。

上面的f1和f2两个式子都是在分段均匀的情况下得来的。
如果是分成两段，那么通过随机交换轴元素，可以从统计上说均匀了。
而分成三段的话，这一点就完全没法保证，
上面的实现是直接取的区间的前两个元素来做轴的。

尽管这个想法没有实践上的任何意义，
用来练习编码还是不错的。
因为3段和2段说起来仅仅是一个数字的不一样，
实现起来才发现要考虑的东西还真有不少。
为了保证2*n的比较次数，我用了O(n)的辅助空间，
也同时引入了大量元素的搬移工作。
