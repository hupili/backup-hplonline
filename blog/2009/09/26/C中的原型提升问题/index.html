C中的原型提升问题

Sep 26th, 2009

(hplonline)2009.9.26

很早看过K&R风格和ANSI风格的函数声明及定义。
一直觉得是习惯问题，才发现这个跟原型提升很有关系。

》原型提升：（32位）
char，short等在运算和传参的时候提升成int型的。
float提升成double型的。

这样做的最初目的无疑是简化编译器的设计，
但是在函数调用上确实存在一点问题。
下面的实验需要将相关函数的实现放在不同的C文件里面，
如果在同一个文件里面，编译器往往能掌握足够的信息，
也就看不出有什么问题了。

》K&R风格：

定义：
void show(f,i)
float f ;
int i ;
{
    printf(“%f,%d\n” , f , i) ;
}
声明：
void show() ;

》ANSI风格：

定义：
void show(float f,int i)
{
    printf(“%f,%d\n” , f , i) ;
}
声明：
void show(float f , int i) ;

》测试代码：

int main(){
    float f = 1.0 ;
    int i = 1 ;
    show(f , i) ;
    return 0 ;
}

用K&R风格的声明及定义，
以及ANSI风格的声明及定义，
在策略上是匹配的，所以没有什么问题。
输出均为：
1.000000,1

》K&R风格定义+ANSI声明：

输出为：
0.000000,0

由于编译器看到的声明是
void show(float f , int i) ;
所以对于f变量不提升为double。

而在实现文件中
void show(f,i)
float f ;
int i ;
由于是K&R风格的定义，
编译器会认为栈中先存放的是提升过的double型，
以及一个int型，总共12个字节。
默认的行为是，先取出前面的8个字节，
按照double转float的规则附给f。
紧接着的4个字节直接放到i变量里面。

由于中间的4个字节在调用端是push的1进去，
按照double数据的表示方法，
前面的8个字节会被看做一个指数负得很多的数，
所以输出的是0。

而打印出来的i为0，是因为这个时候栈上那几个字节为0，
这个值是不定的，根据栈的情况表现不同。

》ANSI风格定义+K&R声明：

输出为：
0.000000,1072693248

由于编译器看到的声明为：
void show() ;
这里没有提供任何类型方面的信息，
所以对f变量进行提升后才传入，
这个时候实际push到栈上的内容有12个字节。

而实现文件中
void show(float f,int i)
由于是ANSI风格的定义，
会认为低端4个字节就是float，
跟着的4个字节是int。

这样等于说把调用端传进来的double(1.0)拆成了两段。

由于double的0-51位表达的是尾数，
而尾数的表达是省略最前面的1的，
也就是后面这些位都是0了（表示1.0000000….，二进制)
（具体参看将二进制实数正规化的资料）
这样一个全0的东西按照float来理解的话，就是0了。

而高四位的内容包含了原来double的符号位和指数和部分尾数。
指数的表达是偏移了2^10-1过后的，（double总共有11为为指数位）
再加上后面尾数位的20个0，
所以这个数字按照int的方式来观察，得到的是
(0 + 2^10-1) * 2^20 = 1072693248

》总结

原型提升是件相当邪恶的事情，
不过只要保证声明和定义的一致，
就没有问题了。。

在这篇的实验中，其实还用到了printf的原型提升。

之前研究printf和scanf对浮点数的处理时，
完全是从汇编的角度看出来的这么个结论。
结果后来发现，10多年前的N本书都讲过这个问题。。。

而同样的东西，从不同的角度来理解一下，确实很有意义。
