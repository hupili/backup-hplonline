Goertzel算法

Jan 1st, 2010

(hplonline)2010.1.1

今天突然又提起了这个名字，
于是决定看个究竟是什么样的东西。

在网上翻了一下，
看了一篇博客，一个网页，一篇paper。。
本来是个很简单的东西，
却把国人粗制滥造的精神体现得淋漓尽致。。。

于是我就一点一点来说吧，
里面用到的公式和图就直接截别人做的了，
这样更加真实地反映我们能从网络得到什么。

该算法的意义：

如果搜一下的话，会看到到处说是DFT的快速计算方法。
然后会提到如DTMF这样的东西。
其实就是两点：
1.有的时候只关心部分频率分量，不需要计算所有分量。
2.希望边读入边计算 ，而不是读取完整一帧后才能计算。

基本原理：


（图来自http://www.dzsc.com/data/html/2008-12-16/75269.html）

这张图还是不错，根据这个意思就知道整个递推过程应该是怎样的。
这个有点像基础算法中提到的多项式求值问题。
这样加乘迭代进行可以减少乘法次数。
但这是次要问题，主要在于可以边输入边计算。

具体的一阶系统如下图：


（图来自http://www.dzsc.com/data/html/2008-12-16/75269.html）

其实就这样两个图，比有些地方直接给出的算式容易理解得多。
而且有些地方还是直接很陡地给出实数计算方法，
让第一次看这个算法的人反映不过来。

第一个猫腻：

大部分人会就此停止。
（原理都清楚了，实现上有什么问题吗？）
但我看到的三份材料都没有指出一个细节问题：
即计算结果和真正DFT的差异问题。

其中两份是完全没有提到。
维库的那篇则是想办法规避掉了。

仔细看上面一阶系统下面的计算过程。
你会发现先输入的是x[3]在x[2]。。。
实际上我们的序列是x[0],x[1],x[2]这样过去的。
如果在每一帧中从后向前输入，
意即要进行存储，便失去了该算法的第二条意义。

那么从x[0]开始输入得到的结果是什么呢？
把式子写出来就会发现，实际上和DFT的定义式是有出入的。
但我看到的大部分地方就直接当DFT上 那个频点的值使用了。

通过推导可以得到像上面系统输出的y[n]实际上是这样的一个东西
y = conj(X[k]) * exp(j * 2 * pi / N * k)          （1）

通过下面的程序验证：

clear ;
clc ;

N = 10 ; %points per frame
k = 4 ; %the frequency to caculate
x = rand(1 , N) ; %randomize a series

X = fft(x) ;
Xd = X(k + 1) %result from ordinary DFT

Wk = exp(-j * 2 * pi / N * k) ;
Xg = 0 ;
for ii = 1:N
Xg = Wk * Xg + x(ii) ;
end
Xg %result from most derivations of goertzel
Xg = conj(Xg * Wk) %corrected result of goertzel

输出：

Xd =
-0.5698 + 0.0650i
Xg =
0.4992 - 0.2823i
Xg =
-0.5698 + 0.0650i

第一个Xg 就是那个系统输出的结果，
第二个Xg 是根据关系式（1）调整过后的结果。
可以看到，后面这个才是真正的DFT值。

那为什么大家都不提这个猫腻呢。。。？
我觉得主要原因是在应用中，关心的是幅度问题。
观察（1）式，实际上乘旋转因子和取共轭都是不影响幅度的。
但是不管怎么说，指出有这个差异，
交给读者去推导也是行的。
不能只字不提，让人以为是对的，
却推很久推不通。。

实系数计算法：

这是来自某篇paper的截图，
要注意他这里用的旋转因子和我们平常定义的不一样，
指数部分刚好是相反数。

那个式子看起来复杂，实际上可以这样来看：
先写出y[n] = WNk * y[n-1] + x[n]对应的一阶系统z域描述，
再把分子分母同时乘一个分母的共轭（乘共轭是常用的实化手段）

分母被实化了之后，整个递推过程就可以做到实数乘法。
但递推完了之后，需要一步的复数计算。
要知道这点，看一下系统的框图就知道了。

其实这篇paper把这些写出来还是不错的，
至少比有些地方直接给出结论容易接受多了。
不过悲剧的是他仍然写错了一个符号。
我怀疑他的WNk的定义是为了标新立异，
结果到后面自己把自己搞囧起了。

我一开始也有个疑问，就是分母实化后，
分子仍然是个复数，怎么可能做到实数计算呢？

但巧妙的就在这个地方，
因为我们不关心中间的y[n-1],y[n-2]等等，
只关心y[n]。
而y[n] 只和w[n] , w[n-1]相关。
又计算w[n]的部分是纯实数的，
于是可以先通过N步的实数递推
得到w[n]和w[n-1]，
然后通过一步的复数计算得到y[n]。

验证程序如下：（接在上面的程序后面）

cosk = 2 * cos(2 * pi / N * k) ;
reg1 = 0 ;
reg2 = 0 ;
reg0 = 0 ;
for ii = 1:N
reg0 = x(ii) + cosk * reg1 - reg2 ;
if ii == N
break ;
end
reg2 = reg1 ;
reg1 = reg0 ;
end

Xg2 = reg0 - exp(j * 2 * pi / N * k) * reg1
Xg2 = conj(Xg2 * Wk)

这里用到了三个寄存器。
循环中那个break相当重要，要不reg1的值就被覆盖了。。
我开始就没注意到这点，发现结果一直对不上。

结果：

Xg2 =
0.4992 - 0.2823i
Xg2 =
-0.5698 + 0.0650i

与之前的数据对比，知道是正确的。
同时也是需要根据（1）式做相应的调整，才是DFT的值。

第二个猫腻：

看到有地方提Goertzel优化算法。

结果仔细一看，并没啥本质的效率改进。
递推过程是必然进行的。
仅仅是在最后从中间结果w[n]和w[n-1]直接推的y[n]的幅度的表达式而已。
而一帧的计算量已经远远大于最后这一点点计算，
连常数级的改进都算不上，怎么能说优化呢。
